# Web App フロー

## 概要

非エンジニアがブラウザで動くWebアプリのプロトタイプを作るためのフロー。
技術選定の負担をゼロにし、「何を作りたいか」だけに集中させる。

このフローは、共通ディスカバリーで「最小解決手段は Web App」と決まった後に使う。

## デフォルト技術スタック

ユーザーに技術の話は一切しない。裏側で以下を使う：

- **フレームワーク**: Next.js (App Router)
- **言語**: TypeScript
- **スタイリング**: Tailwind CSS
- **UIコンポーネント**: shadcn/ui
- **データ永続化**: ローカルJSON or SQLite（プロトタイプ段階）
- **パッケージマネージャ**: npm

## ステップ

### Step 1: 画面と導線の整理

共通ディスカバリーで整理した内容を前提に、画面と操作の流れを具体化する。
一度に全部聞かず、2-3問ずつ進める。

**必須：**
- 最も大事な機能は何か？（1つだけ）
- ユーザーは最初に何を見て、何を操作するか？
- 最低限必要な画面は何か？

**任意（聞けたら）：**
- 参考にしたいサービスやアプリはあるか？
- データの保存は必要か？（ログイン、投稿など）

### Step 2: Web App 要件の整理と確認

整理結果を以下のフォーマットで追記し、ユーザーに確認する：

```
【アプリ名】（仮）
【一言で】〇〇するためのアプリ
【コア機能】
  - 機能A（最重要）
  - 機能B
  - 機能C
【画面構成】
  - 画面1: 〇〇
  - 画面2: 〇〇
```

ユーザーが「OK」と言うまで調整する。

### Step 3: 実装（Codex が実行）

1. `projects/{プロジェクト名}/` に Next.js プロジェクトを作成
2. コア機能から実装開始（最重要機能を最初に）
3. 各画面を実装するごとにコミット
4. 実装中にユーザーへの確認は不要（一気に作る）

### Step 4: 動作確認

1. `npm run dev` でローカルサーバーを起動
2. ユーザーに `http://localhost:3000` を開くよう案内
3. 操作方法を簡潔に説明

### Step 5: フィードバックループ

```
触ってみてどうですか？

- 変えたいところ
- 追加したい機能
- 削除したい機能

があれば教えてください。なければ「完成」と言ってください。
```

フィードバックがあれば修正してStep 4に戻る。
「完成」と言われたら終了。

## 注意事項

- エラーが出た場合、技術的な説明は避け「修正しますね」で対応する
- 見た目は最初からそれなりに整える（shadcn/uiのデフォルトで十分）
- 見た目重視の UI や LP なら `frontend-skill` の利用を検討する
- 認証やデプロイはプロトタイプ段階では扱わない
