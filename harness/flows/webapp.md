# Web App フロー

## 概要

非エンジニアがブラウザで動くWebアプリのプロトタイプを作るためのフロー。
技術選定の負担をゼロにし、「何を作りたいか」だけに集中させる。

このフローは、共通ディスカバリーで「最小解決手段は Web App」と決まった後に使う。
複雑な実装では `planning-with-files` の考え方で、`task-plan.md` と `progress.md` を作って進める。
実運用では `interview-summary.md` を読み、そこから画面、導線、制約を具体化する。
デフォルトでは、画面像と PoC 範囲の確認が終わるまで実装に入らない。
UI の印象がユーザー納得に大きく影響する場合は、実装前に Codex の画像生成で画面イメージを作り、方向性の合意を取ってからコード化する。
このスプリントでは、初回 findings フェーズで `mcp_excalidraw` を使い、初回プロト完成直後に `Understand-Anything` を使う。

## デフォルト実装方針

ユーザーに技術選定を求めず、以下をデフォルトにする：

- **フロントエンド**: HTML / CSS / JavaScript
- **バックエンド**: 必要な時だけ Node.js
- **API処理**: 外部 API 連携やバックエンド仲介が必要な時だけ Node.js で追加
- **データ永続化**: プロトタイプ段階ではローカルJSON などの軽量方式を優先

最初から大きなフレームワークを前提にしない。
ただし、画面数や状態管理が増えて vanilla では扱いづらい場合だけ、最小限の構造化を検討する。

## デフォルトデザイン方針

Web App のベースデザインは、現行の NTT DATA ブランドを参照した青系テーマを基準にする。

- ベースカラーは NTT DATA の現行公式サイトや現行 CI の見た目に沿ったブルー基調
- 白背景と青系アクセントを軸に、信頼感と視認性を優先
- 装飾より情報の見やすさを優先

色の厳密な数値が公式文書で明示されていない場合は、現行公式サイトの見た目を基準に近いトーンで合わせる。
旧ブランドではなく、現行ブランドを優先する。

## ステップ

### Step 1: 実装用ファイルの初期化

複雑なアプリでは、実装に入る前に以下を作る。

- `interview-summary.md`
- `task-plan.md`
- `progress.md`
- 必要なら `findings.md`
- 必要なら `findings-diagrams/`

`findings.md` は調査や比較、仕様の迷いがある時だけ使う。
`findings-diagrams/` は図で共有した方が早い時だけ使う。

ここで判断する：

- スコープが複雑なら `planning-with-files` を有効にする
- 単純な PoC なら `interview-summary.md` と `project-requirements.md` から始める

### Step 2: 画面と導線の整理

`interview-summary.md` と共通ディスカバリーで整理した内容を前提に、画面と操作の流れを具体化する。
一度に全部聞かず、2-3問ずつ進める。

**必須：**
- 最も大事な機能は何か？（1つだけ）
- ユーザーは最初に何を見て、何を操作するか？
- 最低限必要な画面は何か？

**任意（聞けたら）：**
- 参考にしたいサービスやアプリはあるか？
- データの保存は必要か？（ログイン、投稿など）

次が曖昧な場合は実装に進まない：

- 最も大事な機能
- 最初の操作導線
- 最低限必要な画面数

### Step 3: Web App 要件の整理と確認

整理結果を以下のフォーマットで追記し、ユーザーに確認する：

```
【アプリ名】（仮）
【入力】interview-summary.md
【一言で】〇〇するためのアプリ
【コア機能】
  - 機能A（最重要）
  - 機能B
  - 機能C
【画面構成】
  - 画面1: 〇〇
  - 画面2: 〇〇
【API / バックエンド】
  - 必要 / 不要
  - 必要なら何を仲介するか
【デザイン方針】
  - NTT DATA 青系テーマで進める
【UIイメージ生成】
  - 必要 / 不要
  - 必要なら、生成する画面と確認したい観点
```

ユーザーが「OK」と言うまで調整する。
ここで合意するのは「PoC で作る範囲」までであり、将来機能までは詰めない。

### Step 4: UIイメージ生成 checkpoint（必要な場合）

次の場合は、実装前に Codex の画像生成でファーストビューまたは主要画面のイメージを作る：

- UI の第一印象が PoC の納得感に直結する
- 画面密度、情報配置、ブランド感の解釈に迷いがある
- LP、ダッシュボード、業務画面など、見た目の期待値が成果物評価に影響する

生成する画像は、最終デザインではなく方向性確認用として扱う。
確認する観点は以下に絞る：

- 情報の優先順位
- 画面密度
- 色・トーン
- 最初に操作したくなる導線

ユーザーが方向性に納得したら Step 5 に進む。
軽微な UI や内部ツールで、要件から画面像が明確な場合はこの Step を省略してよい。

### Step 5: 実装（Codex が実行）

1. `projects/{プロジェクト名}/` にフロントエンドを作成する
2. API 処理やバックエンド仲介が必要な場合だけ Node.js を追加する
3. コア機能から実装開始する（最重要機能を最初に）
4. 重要な判断、`interview-summary.md` からの解釈、調査結果、テスト結果は `progress.md` や `findings.md` に残す
5. 実装中にユーザーへの確認は原則不要

ただし、以下の分岐が出た時だけ止まる：

- 合意済みスコープを超える機能追加が必要になった
- 保存や認証など、前提を変える判断が必要になった
- UI ではなく業務フロー自体を見直す必要が出た

### Step 6: 動作確認

1. `npm run dev` でローカルサーバーを起動
2. ユーザーに `http://localhost:3000` を開くよう案内
3. 操作方法を簡潔に説明

### Step 7: 初回 findings と図の整理

1. 初回プロトで分かったことを `findings.md` に整理する
2. `mcp_excalidraw` を使って、少なくとも 1 つ図を作る
3. 図の候補は次から選ぶ
   - 画面遷移
   - データフロー
   - コンポーネント構成
   - 改善案の比較図
4. `mcp_excalidraw` が使えない場合は、Figma / FigJam の図生成または Mermaid を代替として使う

### Step 8: 初回プロト完成後の構造理解

1. 初回プロトが動いたら `Understand-Anything` を使う
2. 構造理解、依存関係、影響範囲を整理する
3. 次のフィードバック反映で壊しやすい箇所を `findings.md` または `progress.md` に残す

### Step 9: フィードバックループ

```
触ってみてどうですか？

- 変えたいところ
- 追加したい機能
- 削除したい機能

があれば教えてください。なければ「完成」と言ってください。
```

フィードバックがあれば修正してStep 5に戻る。
「完成」と言われたら終了。

## 注意事項

- エラーが出た場合、技術的な説明は避け「修正しますね」で対応する
- 見た目は最初から整えるが、NTT DATA の現行ブルー基調を崩さない
- UI の納得感が重要な場合は、実装前の画像生成を優先する。ただし、画像を仕様書として固定せず、実装時は使いやすさを優先して調整する
- API 処理やバックエンドが必要な場合は Node.js を使う
- 見た目重視の UI や LP なら `frontend-skill` の利用を検討する
- `mcp_excalidraw` は最初の findings フェーズで使うが、要件定義や実装そのものの代替にはしない
- `Understand-Anything` は初回プロト完成直後から使い、実装前の手段選定には使わない
- 図は `findings.md` を置き換えるものではなく、文章の補助として残す
- 認証やデプロイはプロトタイプ段階では扱わない
- `interview-summary.md` が曖昧でも、勝手にアプリの種類やスコープを膨らませない
